daily-trend-viewer — 모드1(태그/카테고리 통합) + 모드2(구독 관리 통합) Implementation Notes

작성 시작: 2026-07-07

요청 원문

모드1) 카테고리/태그 지정 기능을 전 플랫폼(유튜브/쇼츠/AI영상/릴스/트위터/스레드/틱톡)에 통합 — 현재 유튜브/쇼츠만 카테고리·태그 기반이고 나머지는 구독계정 방식.
모드2) 구독자/관심 사용자 등록 후 확인하는 기능(현재 릴스/트위터/스레드/틱톡 방식을 통합 검토).

플랫폼별 현황 (구현 전 조사)

플랫폼기존 방식모드1 목표실현 가능성
유튜브/쇼츠카테고리+자유검색-이미 충족, 변경 없음
AI 영상 탭고정 쿼리 세트-이미 카테고리성, 변경 없음
틱톡트렌딩+구독계정키워드 검색가능 — tikwm /api/feed/search 실제 호출로 검증 완료
인스타 릴스구독계정해시태그 검색가능(추정) — tags/web_info, 기존 세션쿠키 재사용. 응답 스키마는 문헌 기반 추정 파싱(라이브 미검증 — 유효 세션쿠키 없음)
X(트위터)구독계정(syndication)키워드 검색불가 — syndication API는 프로필 타임라인 전용, 비인증 검색 엔드포인트 없음
스레드구독계정(GraphQL doc_id 추측)키워드/태그 검색불가 — 공개 태그 검색 GraphQL doc_id 미상, 기존 프로필 조회도 불안정한데 확장은 리스크 가중
[설계 결정] 2026-07-07
모드1은 "전 플랫폼 100% 균일 구현"이 아니라 기술적으로 실현 가능한 2개 플랫폼(틱톡 키워드, 릴스 해시태그)만 신규 구현하고, X·스레드는 서버 API 검색 없이 프론트엔드 링크아웃(입력한 키워드로 x.com/search / threads.net/search를 새 탭으로 여는 버튼)만 제공한다.
왜: X는 비인증 검색 API가 아예 없고(신드케이션=프로필 전용), 스레드는 기존 프로필 조회조차 doc_id 추측 기반으로 불안정한데 검색까지 확장하면 깨지기 쉬운 코드가 배가된다. "요청 이상으로 확장하지 않는다"는 단순성 원칙보다, 여기서는 반대로 안 되는 것을 억지로 만들지 않는다가 핵심 — 불안정한 추측성 구현보다 명확한 링크아웃이 사용자에게 더 정직하다.
[설계 결정] 2026-07-07
모드2는 신규 플랫폼(예: 유튜브 채널 구독) 확장 없이, 기존 4개 플랫폼(릴스/X/스레드/틱톡)의 계정 관리 프론트엔드 코드 중복 제거로 스코프를 좁힌다.
왜: 사용자 원문이 모드1은 "모든 플랫폼에 통합"이라고 강도 높게 명시한 반면 모드2는 "통합 검토"라고만 되어 있어 요구 강도가 다르다. 백엔드는 이미 ACCOUNT_SOURCES 딕셔너리로 4개 플랫폼이 통일돼 있고, 실제 격차는 프론트엔드에 renderAccountChips/renderAccChips/renderTtChips 식으로 유사 렌더 함수가 중복돼 있는 것 — 이걸 공통 함수로 합치는 게 "통합 검토"의 실질적 답이다. 유튜브에 신규 채널-구독 기능을 얹는 건 명시되지 않은 대형 신규 기능이라 스코프 아웃.
[트레이드오프] 2026-07-07
릴스 해시태그 검색은 IG 세션 쿠키가 있어야만 동작(기존 계정 릴스 조회와 동일 제약). 현재 워크트리에 저장된 유효 세션 쿠키가 없어(ig_session.json 부재) 라이브 응답 스키마 검증을 못 했다. 파싱은 인스타그램 웹 내부 API의 공개적으로 알려진 tags/web_info 응답 구조(data.top/recent.sections[].layout_content.medias[].media)를 기반으로 방어적으로(.get() 체인) 작성한다.
[미결 질문] 2026-07-07
릴스 해시태그 파싱이 실제 스키마와 다를 경우, IG 로그인 세션을 등록한 뒤 /api/reels?tag=... 실호출로 재검증이 필요하다. 세션 쿠키 로그인 전까지는 "빈 배열 반환 + 에러 없음"으로 안전하게 저하(degrade)되도록 구현.

모드1(태그 통합): 틱톡 /api/tiktok?q= 키워드 검색, 릴스 /api/reels?tag= 해시태그 검색 신규 구현. X/스레드는 설계대로 프론트엔드 링크아웃(window.open)만 제공.
