# 반복 문제해결 루프 (rapid-iteration-loop) — 로그드리븐 BTS

> fable-team 문제해결의 **기본 방법론**. "큰 수정"(재설계)이 아니라 **빠른 테스트 → 문제 발견 → 수술적 대응 → 재테스트** 반복으로 작업리스트를 완성한다.
> 출처: BYZ-Agents 로그드리븐 방법론 일반화 (사용자 지침 2026-07-04). 사용자 전역 CLAUDE.md 「🔁 로그 드리븐 = 아키텍처 대조 루프」의 fable-team 판.

## 핵심 원칙 (깊게 생각하지 말 것)

- **큰 수정(재설계) 금지** — 아무 소용없다. 테스트로 발견한 문제에만 수술적으로 대응한다.
- **작업리스트를 따라 완성** — "완성해야 할 것" 목록을 따라가며, 발견되는 문제를 하나씩 처리한다. 문제를 미리 상상해 크게 고치지 않는다.
- **깊게 고민하지 말고 반복** — 많은 테스트 → 많은 문제 발견 → 빠른 반복이 살길이다.
- **로그는 얼마든 심는다** — 고민 말고 로그 심고 재테스트. 수정이 증명되면 심은 로그 정리.
- **프론트 결과 + 백엔드 동작(BTS) 함께 확인** — 프론트에 나온 결과와 백엔드에 쌓인 동작(BTS)의 괴리에서 문제가 드러난다.

## 문제해결 7원칙 (기본 지침 — 모범사례 박제)

> 출처: 2026-07-14 v6-realtime-live "우아한 마무리" 세션에서 실증된 패턴을 일반 지침으로 박제. 특정 케이스 과적합이 아니라 fable-team이 일 잘하는 일반 방법이다.

1. **빅뱅 금지 — 1스텝씩 수렴**: 큰 통합 설계를 한 번에 하지 않는다. **원인 판별을 먼저 1스텝**으로 하고, 원인이 확정된 뒤에만 fix 설계로 넘어간다. **여러 이슈를 병렬로 조사하는 것도 빅뱅(오케 레벨)** — 딱 한 문제만 붙잡는다(수집용 checker 병렬은 무방, "이슈" 병렬이 금지). **설계 원자성 ≠ 커밋 원자성**: architect/DA는 "무엇을 고칠지"(설계 — fail-closed barrier로 상호의존이어도 단일 계약 가능)만 정하고, **몇 커밋으로 랜딩할지는 오케 책임**이다 — 설계가 단일 계약이어도 Tidy First로 구조(추출·정리) 선커밋 → 행위(로직) 후커밋 분할한다. 커밋 크기 통제 실패는 오케 잘못이지 architect/DA 탓이 아니다.
2. **계측으로 원인을 좁힌다 (추측 금지)**: 가설이 여러 갈래(A/B/C)면 **행위변경 없는 계측(로그)** 을 심어 데이터로 판별한다. 코드만 읽고 "아마 A일 것"으로 넘어가지 않는다.
3. **의심 가설을 실증으로 반박한다**: 의심되는 가설도 코드/로그/데이터 실증으로 **반증**해 틀린 방향을 조기 제거한다 — 단순 뒤로 미루기(deprioritize)가 아니라 능동 falsify.
4. **체커부터 — 실제 로그+짧은 재현 후 기획**: ①의심지점 로그 심기 → ②짧은 재현 1회 → ③체커가 실행로그+데이터 수집·정리 → ④그 실증 기반으로만 기획. **체커 수집 전에 기획으로 건너뛰지 않는다**(정적 코드리딩만으로 architect 설계 착수 금지).
5. **architect ↔ DA 직접 approve loop (오케 중계 없이)**: architect가 설계하면 DA에게 **직접 send**하고, DA 피드백을 **직접** 받아 반영·재발주를 둘이서 수렴한다. 오케는 중간 릴레이를 받지 않고 **APPROVE 확정된 최종본만 1회** 공유받는다(§역할 체인 · SKILL.md「직접 approve loop」).
6. **과적합·엣지케이스를 DA가 반박한다**: DA는 단순 승인기가 아니라 적대검증 게이트다. **하드코딩 상수의 과적합 여부**, **silent-fallback형 엣지케이스**(예: missing 이벤트 시 영구 고착)를 반드시 반박·검증한다.
7. **완성 = 라이브 반응 직접 관찰**: 코드 작성·유닛 통과는 완성이 아니다. **실제 라이브 테스트로 반응을 직접 보며** 개발해야 완성이다(운영규율 #3 close=라이브 JSONL과 동일 축). 증거 형식은 **아티팩트 성격에 맞게** — 라이브 UI/서비스는 trusted 클릭·실입력 관측 JSONL, 라이브 접점이 없는 아티팩트(라이브러리·CLI·배치)는 그에 맞는 실행 관측 산출물(신선한 실제 동작 증거). **"1회 돌려 판정" 금지** — 짧은 반복(1회 실행 → 로그 대조 → 판정 → 다음)으로 여러 번 대조하고, fix 안에 국소화 로그를 심어 매 실행마다 대조한다 — FAIL 시 여러 변경 블록 중 범인을 특정할 수 있어야 revert도 국소로 끝난다.

## 역할 체인 (순서 고정)

```
체커(로그/코드/스펙 3자대조 실증)
  → architect(실증 기반 작게 기획) ↔ DA(과적합·엣지 반박, 직접 approve loop)
  → impl(수정)
  → 테스터(라이브 재현)
  → 체커(수집) → 확인
  → PASS면 다음 이슈 / FAIL이면 원인 재분석
```

- **한 번에 한 이슈만** 이 체인을 탄다(원칙 1). 다음 이슈는 현 이슈가 PASS로 닫힌 뒤.
- architect↔DA 구간은 **직접 send 루프**(원칙 5) — 오케는 최종 APPROVE만 수신.

## 계측 후 정리 (진단 로그 위생)

- 진단용 임시 로그는 **미커밋**, 원인 확정 후 반드시 **제거 확인**한다.
- 커밋 스택에 진단 로그가 새지 않게 **diff 직독으로 검증**(오케 또는 게이트가 커밋 전 대조). 계측 커밋은 사용자 승인 게이트(운영규율 #4).

## 루프 (반복 — 이게 살길)

```
1. 테스트 (많이)     → byz-e2e / omo qa / ft-tester. 프론트 결과 + 백엔드 BTS 동작 확인
2. 문제 발견         → 프론트 결과 ↔ 백엔드 BTS 괴리. 많은 테스트로 많은 문제 노출
3. 로그·컨텍스트 파악 → ft-checker(sonnet)가 로그·BTS·아키텍처 문서 빠르게 파악 (병렬 다수, 단말성)
4. 문제 확인 (게이트) → ft-architect(기획, 스펙/아키텍처 대조 원인규명) + ft-da(DA approve loop 적대검증)
5. 수술적 수정       → ft-implementer surgical. 큰 수정 아님, 발견된 문제만
6. 재테스트          → 1로. 통과할 때까지 반복
7. 증명 후 로그 정리  → 수정이 증명되면 심은 로그 제거
```

## 3자 대조 (로그드리븐 = 아키텍처 대조 루프)

로그를 보는 게 끝이 아니다. **①로그가 가리키는 코드 ②스펙·아키텍처·문서(설계의도) ③코드 동작성 ↔ 로그** — 3자 대조로 설계의도 vs 버그를 명확화한다. 땜빵이 아니라 설계대응(필요 시 아키텍처 변경까지). 3자 대조가 근본을 드러낸다.

## 도구 매핑

| 단계 | fable-team 워커/크루 | 브레인 |
|------|---------------------|--------|
| 테스트·라이브 e2e·BTS | **byz-e2e 크루** / **omo qa 크루** / **ft-tester** | sonnet-5 |
| 로그·BTS·아키텍처 문서 파악 | **ft-checker** (병렬 다수, 단말성) | sonnet-4-6 |

| 적대검증 게이트 | **ft-da** (approve loop) | codex-5.6-sol / grok-4.6 |
| 수술적 수정 | **ft-implementer** | opus-4-8 |

- 로그·BTS 파악은 **sonnet(ft-checker)** 로 빠르게 분산 — 오케/기획 세션에 로그 본문을 싣지 않는다.
- 테스트 실행은 프로젝트에 설치된 크루로 선택: byz-e2e(라이브 e2e), omo qa, 또는 ft-tester(범용).

## 금지

- 스파게티·기존구조 꼬임·타시스템 마비. **수술적만**.
- **"단위 테스트/jest green = 완료" 아님** — 라이브 e2e + 백엔드 BTS 동작 확인이 완료 게이트.
- 문제를 상상해 미리 크게 고치기 — 테스트로 실제 발견된 것만 대응.

## 안티패턴 체크리스트 (하지 말 것 — 7원칙의 반대상)

발주·기획·검증 전 이 목록과 대조한다. 하나라도 해당하면 멈추고 교정한다:

1. **추측으로 큰 수정** — 계측 없이 "아마 A일 것"으로 재설계 착수(원칙 2·3 위반).
2. **여러 이슈 병렬 조사 (오케 레벨 빅뱅)** — 한 번에 여러 문제를 동시 조사·설계(원칙 1 위반). 수집 checker 병렬은 예외, "이슈" 병렬이 금지.
3. **체커 수집 전 기획 착수** — 실제 실행로그·짧은 재현 없이 architect가 설계 시작(원칙 4 위반).
4. **오케가 approve loop 중간 릴레이** — architect↔DA 사이를 오케가 판단·재구성해 중계(원칙 5 위반). 오케는 최종 APPROVE만 수신. *(단 오케가 tmux 아닌 Legacy 경로면 판단·지연 0의 pass-through 릴레이는 허용 예외 — 순수 전달만.)*
5. **DA 단순 승인(반박 없이)** — 과적합 상수·silent-fallback 엣지케이스 검증 없이 APPROVE(원칙 6 위반).
6. **유닛 통과 = 완성** — 라이브 반응 관찰 없이 done 전환(원칙 7·운영규율 #3 위반).
7. **하류 수용(근원 버그를 계약으로 승격)** — 소비자(consumer)에서 잘못된 타입/형상을 받아들여 우회하지 말고 **근원(origin)에서 고친다**. 예: mapper가 잘못된 타입을 받으면 mapper에 방어코드를 넣는 게 아니라 그 타입을 만든 곳을 고친다. *(단 외부·신뢰경계(untrusted 입력·버전 호환 경계)에서의 소비자 검증은 정당 — 근원이 통제 가능한 내부일 때 이 원칙이 적용된다.)*
8. **빅뱅 커밋 + 1회 판정 라이브** — 다블록 변경을 단일 커밋으로 랜딩(FAIL 시 통째 revert — 원칙 1 커밋 분할 책임 위반)하고, 라이브 검증을 1회 돌려 PASS/FAIL만 판정(원칙 7 위반 — 국소화 로그 없이는 범인 특정 불가). 실사례: 520+/52- 6블록 단일커밋(82e1399e).
