# 오케스트레이션 케이스 하네스 — 4그룹 · 8케이스 · 모듈 카탈로그

> SSOT: 케이스 분류·모듈 조합·이름 체계·킥오프 템플릿의 원본. 구 `deployment-patterns.md`(P-코드 7종)를 통합 재편 — P-코드는 별칭으로 유지(하위호환).
> 설계 원리: **하네스 = 모듈의 조합**. 상시 모듈은 모든 케이스에 자동 포함되고, 케이스는 선택 모듈의 기본 조합을 정의하며, 스위치로 조정한다. 조합 판단은 오케 LLM 자율 — 이 문서는 큰 틀(대표 케이스와 기본값)만 제공한다.

## 모듈 카탈로그

### 상시 모듈 (매 케이스 자동 — 선택 불요)

| 모듈 | 내용 (정본) |
|------|------------|
| 문제해결 7원칙 | 빅뱅금지·계측·실증반박·체커선행·직접승인루프·DA반박·라이브관찰 (`rapid-iteration-loop.md`) |
| 계측 위생 | 진단 로그 미커밋·확정 즉시 제거·diff 직독 (`rapid-iteration-loop.md` §계측 후 정리) |
| mbox 파일 큐 | 세션간 본문 = `ft-mbox.sh send`(파일 큐 + doorbell), 수신 = `recv`(READ 규약). 검증 송신 4단계는 인터랙티브 예외 (COMM-GUIDE, 스폰 시 주입) |
| 센티널 보고 | 워커 완료 = 파일 센티널, 오케 poll 수거 (`context-management.md`) |
| 증류·편집 게이트 | ctx 임계 증류 + 오케 코드 2파일 제한 (`orchestration-gate.md`) |

### 선택 모듈 (케이스가 기본 조합을 정의)

| 모듈 | 워커/장치 | 역할 한 줄 |
|------|----------|-----------|
| **수집** | checker ×N | 로그·코드·문서 병렬 서치 (읽기전용·단명) |
| **진단** | analyst | 로그↔코드↔스펙 3자대조, DIAGNOSIS + ESCALATE 판단 |
| **설계** | architect | 실증 기반 해결 설계 → 설계 파일 |
| **적대검증** | da (+da2) | 과적합·엣지 반박 게이트 — 강도: 승인루프(게이트) / 후행리뷰(비게이트) / 생략 |
| **승인루프** | architect↔DA | 직접 send 수렴, 오케는 최종 APPROVE만 (7원칙 §5) |
| **구현** | implementer | 설계 SSOT 기반 수술적 수정 |
| **실증** | tester (+tester2) | 라이브 반응 관찰 + 증거 산출 (7원칙 §7) |
| **빠른확인 루프** | 오케 즉결 지시 | "관찰이 곧 판정" 건의 규명 가속 — 실증팩→설계 송부 (`rapid-iteration-loop.md` §다이렉트 확인 루프) |
| **이중오케** | 메인오케+수행오케 | 장기 트랙 2계층 — 메인=HIL·태스크풀·스톨감시 / 수행=워커 지휘 (SKILL.md §2계층 확장) |
| **기록비서** | ft-pm-memory | 흐름 기억·원장·cairn 대행·증류 브리핑 (프로젝트당 1개) |

## 4그룹 · 8케이스 결정표

분류 신호 → 그룹 → 케이스. 모호하면 킥오프 질문에 2안 병기.

| 그룹 | 케이스 | (구 P-코드) | 분류 신호 | 기본 모듈 조합 | DA 강도 |
|------|--------|-------------|-----------|----------------|---------|
| **규명** (변경 없음) | **조사보고** | P-ANLZ | "왜·원인·확인·조사·구조 파악" | 수집×N → 진단. 보고로 종결 | 없음 (선택: 후행리뷰) |
| **규명** | **빠른확인** | (신설) | 관찰 1~2회로 가설 판별 가능·경합가설 계측 1줄 판별 | 빠른확인 루프 단독 → 실증팩 → (필요 시 수정 그룹 케이스로 승격) | 없음 |
| **수정** (기존 동작 고침) | **원인수정** | P-BUG | 에러 로그·재현 절차·회귀 보고 | 수집 → 진단 → [자명하면 설계 생략] → 구현 → 실증 → 적대검증 | 승인루프 |
| **수정** | **긴급직행** | P-HOT | "지금·프로덕션·장애" + 원인 기지 | 구현+실증 동시 직행 | 후행리뷰 1회 |
| **수정** | **반복수선** | (신설) | 이슈 다수 순차·반복 라이브 press·장기 트랙 | rapid-iteration 루프 전체 + 기록비서. 스위치: 이중오케 권장 | 승인루프 (건별) |
| **구축** (새 것·구조) | **신규구축** | P-FEAT | "추가·구현·만들어" + 스펙 | 설계 → 승인루프 → 구현 → 실증 | 승인루프 |
| **구축** | **구조정리** | P-REFAC | "정리·구조 개선·중복 제거" (행위 불변) | 수집(영향 범위) → 설계 → 구현 → 실증(회귀 동일성) | 승인루프 |
| **판정** (완료물 검증) | **검증판정** | P-VRFY | "돌려봐·확인해줘·PASS 여부" | 실증(+병렬) → 적대검증 | 후행리뷰 |
| — (그룹 밖) | **직접처리** | P-DOC | *.md·설정·오타 (코드 ≤2파일) | 워커 미스폰 — 오케 직접 (게이트 허용 범위) | 없음 |

### 스위치 (케이스에 얹는 조정 — 킥오프 옵션 또는 실행 중)

| 스위치 | 켜는 조건 |
|--------|----------|
| `+이중오케` | 진입점 3개 — ①킥오프에서 사용자 1클릭(반복수선 기본 권장) ②중간: 오케가 **활성 이슈 3건+ 병행** 감지 시 제안→사용자 승인 ③사용자 직접 트리거("장기전 전환"/"2계층으로"). 전환 절차(신규 세션=메인오케 레디, 기존 오케=수행오케 전환, 워커 재공지)는 SKILL.md §2계층 확장 |
| `+기록비서` | 코드 변경 케이스(수정·구축)는 기본 켬 / 규명·판정·직접처리는 끔 |
| `+빠른확인 선행` | 수정·구축 케이스인데 원인/전제 판별이 관찰 1~2회로 가능할 때 |
| `+이중판정` (da2) | 고위험·광범위 변경 — 이종 브레인 교차 |
| `+병렬실증` (tester2) | 검증 케이스가 많아 이분할이 이득일 때 |
| DA 강도 전환 | 승인루프(게이트) ↔ 후행리뷰(비게이트) ↔ 생략 — 위험도 기준 오케 자율 |

### 승격·증원 (실행 중 케이스 전환 — 자동 신호)

- 진단 `ESCALATE_TO_ARCHITECT: yes` → 설계 투입 (원인수정 → 신규구축급 승격)
- 실증 `FAIL` 동일 케이스 2회 → 진단 투입 (긴급직행/검증판정 → 원인수정 승격)
- 빠른확인 실증팩이 큰 수정을 가리킴 → 수정 그룹 케이스로 승격 (fix는 1줄이라도 설계+DA 경유)
- 증원은 같은 세션 선택 브레인 사용(재질문 금지), 사실만 state.md 1줄 기록.

## 운영 규칙

1. **필요 모듈만 스폰** — 케이스 기본 조합이 곧 스폰 목록. 없는 워커는 대기 스폰조차 금지.
2. **해산** — 단명 워커(수집)는 done 즉시 kill, 상주 워커는 파이프라인 종결 시 오케 명시 종료.
3. **케이스 이탈 허용** — 킥오프에서 사용자가 조정 가능. 조정 결과가 그 피처의 형상(state.md 기록). 카탈로그는 추천 기본값이지 락이 아니다.
4. **한 번에 한 케이스** — 1이슈 1케이스(7원칙 §1). 다음 이슈는 현 케이스 close 후.

## 킥오프 질문 템플릿 (공통 1형)

오케가 분류 직후 AskUserQuestion 1개 발사. 공통 스펙: `multiSelect: false`, `header: "배치"`, 첫 옵션에만 ` (Recommended)`, label ≤30자, 엔터 = 즉시 킥오프.

```yaml
question: |
  문제: <한 줄>
  케이스: <케이스명> — <그룹> (<분류 이유 한 구>)
  산출: .fable-team/features/<slug>.md   # 직접처리는 "직접 처리"
  연동: <워크트리·cairn 1줄 또는 "없음">
options:
  - label: "<케이스명> 기본 (Recommended)"
    description: "<기본 모듈 체인 → 표기> + DA <강도>"
  - label: "+ <스위치 1>"            # 위 스위치 표에서 이 케이스에 유효한 것 최대 2개
    description: "<켜는 조건 한 줄>"
  - label: "<축약 또는 인접 케이스>"   # 모호 분류면 2순위 케이스 병기
    description: "<발동 조건 한 줄>"
  - label: "직접 조합"               # ask_other_builtin이면 생략
    description: "모듈·순서·DA 강도 자유 입력 (1회 재확인 후 킥오프)"
```

- 케이스별 고정 4옵션 각본은 두지 않는다 — 위 표(기본 조합·스위치)에서 오케가 그 문제에 유효한 옵션을 조립한다(LLM 자율).
- 직접처리만 예외: 옵션 = 직접 처리(Recommended) / 구현 위임(코드 3파일+) / 신규구축 승격(동작 변경 수반).
