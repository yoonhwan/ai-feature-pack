# fable-team 피처 설계 인터뷰 (매 피처 시작 시)

설치가 끝난 팀에서 새 피처/태스크를 시작할 때 오케스트레이터가 진행하는 절차. 목적: **무엇을 할지**를 최소 비용으로 받고, **프로젝트가 이미 가진 자산(스킬·플러그인·하네스·도구)을 서치해 추천 기반으로** 설계를 확정한다.

## 1. 피처 입력 인터뷰 (무조건 — 생략 금지)

**AskUserQuestion으로 반드시 받는다** (SKILL.md 부팅 시퀀스 2). 이전 대화에 태스크가 보여도 인터뷰를 건너뛰고 파이프라인을 시작하지 않는다. 선택지:

- **① ② 컨텍스트 추천**: 기존 대화에서 추출 가능한 태스크 후보 최대 2개를 각 한 줄로 요약해 추천 옵션으로 제시 (후보 1개면 ①만).
- **③ 내용 추가**: 직접 입력 — 한 줄 메시지("무엇을 만들/고칠까요?" 한 문장이면 충분) 또는 스펙/이슈/로그 **파일 경로**. 파일이면 checker 워커에게 요약시켜 컨텍스트 노트로 만든다 (오케스트레이터가 직접 정독하지 않는다 — 컨텍스트 절약).
- **④ 채팅에서 이어하기**: 인터뷰를 닫고 대화 흐름에서 피처를 구체화 — 구체화가 끝나면 확정 내용을 한 줄로 재확인 받은 뒤 §2로 진행.

후보 추출이 불가하면 ③④만 제시. 어떤 경로든 **사용자 응답 없이 다음 단계 진행 금지.**

## 2. 프로젝트 자산 서치 (추천 재료)

병렬로 수행 (checker 워커 또는 직접 Glob):
- `<project>/.claude/skills/*/SKILL.md` — 프로젝트 스킬 (name/description만)
- `<project>/.claude/agents/*.md` — 기존 커스텀 에이전트
- `<project>/CLAUDE.md`, `AGENTS.md` — 프로젝트 지침·하네스 규약
- `Makefile`, `package.json`(scripts), `pyproject.toml` — 빌드/테스트 러너
- `.mcp.json`, 설치된 플러그인 — 사용 가능 MCP/하네스

## 3. 추천 기반 설계 인터뷰

수집 결과로 **추천안을 먼저 제시**하고 확인받는다 (열린 질문 금지, 추천+대안 구조):

```
이 피처에 추천하는 구성:
- 파이프라인: [표준 6단계 | 축약(확인→구현→테스트) | 확인만]
- 활용 자산: implementer가 <스킬X> 사용, tester는 `make test` 러너 (발견된 것 기반)
- 워커 증감: checker N명 병렬 (대상 파일 수 기반)
- DA 강도: [approve loop 2라운드 | review만 | 생략]
진행할까요? (조정 가능)
```

- 추천 근거는 §2에서 실제 발견된 자산만 사용 — 없는 스킬/러너를 추측으로 넣지 않는다.
- 사용자가 조정하면 해당 항목만 바꿔 재확인 1회.

## 4. 설계 확정 → 킥오프

- **컨펌 게이트 (SKILL.md 부팅 시퀀스 4~5)**: 확정 전 실행 준비 프리뷰(부팅 보드 — 피처/형상/로스터/산출물 경로/라운드 한도)를 보여주고 사용자 승인을 받는다. **컨펌 없이 워커 스폰 금지.**
- 확정 내용을 `<project>/.fable-team/features/<slug>.md`에 기록 (피처 한 줄, **파이프라인 형상(파이프라인 + DA 강도)**, 자산 배정, 성공 기준) — 형상은 킥오프 시 state.md frontmatter(`pipeline`/`da`)로 복제돼 세션 복원의 분기 기준이 된다(context-management §4).
- 이 파일이 planner에게 넘어가는 1차 컨텍스트가 된다.
- 이후 `orchestration-playbook.md` 파이프라인 시작.
